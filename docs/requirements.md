# Требования к системе volunteer-desk-site

## Назначение системы

Веб-сайт для приёма заявок на волонтёрскую помощь и управления ими. Позволяет гражданам подать заявку онлайн, а администратору — просматривать, принимать или отклонять заявки, назначать волонтёров и формировать печатные версии для передачи исполнителям.

---

## Роли

| Роль | Описание |
|------|----------|
| **Заявитель** | Анонимный посетитель сайта. Заполняет форму заявки на помощь. |
| **Администратор** | Авторизованный пользователь. Управляет заявками через административную панель. |
| **Волонтёр** *(v2)* | В будущей версии — может оставлять отметку о выполнении или подавать заявку на участие. |

---

## Функциональные требования

### 1. Главная страница (публичная)

- Название и описание проекта.
- Информация о том, для кого предназначена помощь.
- Краткое описание процесса: заявитель → администратор → волонтёр.
- Кнопка «Оставить заявку» с переходом на форму.

---

### 2. Форма подачи заявки

Поля:

| Поле | Тип | Обязательное |
|------|-----|:---:|
| ФИО | текст | да |
| Район | выбор из справочника | да |
| Описание необходимой помощи | большое текстовое поле | да |
| Количество необходимых волонтёров | число | нет |
| Комментарий | текстовое поле | нет |

После успешной отправки заявитель видит подтверждение: «Спасибо! Ваша заявка принята и будет рассмотрена администратором.»

Требования к форме:
- Валидация обязательных полей на стороне клиента и сервера.
- Защита от спама (Google reCAPTCHA или аналогичное решение).

---

### 3. Статусы заявки

| Статус | Описание |
|--------|----------|
| **Новая** | Заявка только что создана, ещё не обработана. |
| **Принята** | Администратор подтвердил заявку. |
| **Отклонена** | Администратор отказал в принятии заявки. |

Переходы статусов инициируются администратором вручную через кнопки в карточке заявки.

---

### 4. Административная панель

Доступ только для авторизованных пользователей (логин + пароль).

**Список заявок** — таблица со столбцами:
- Номер заявки
- Дата создания
- ФИО
- Район
- Статус

**Поиск и фильтрация** *(v1 — архитектурная закладка, v2 — полноценный UI)*:
- Поиск по ФИО и по названию района.
- Фильтрация по статусу: Новые / Принятые / Отклонённые.

**Карточка заявки** — детальный просмотр с полями:
- ФИО
- Район
- Описание необходимой помощи
- Количество необходимых волонтёров
- Комментарий
- Дата создания
- Текущий статус

Кнопки действий в карточке:
- **Принять заявку** → статус меняется на «Принята».
- **Отклонить заявку** → статус меняется на «Отклонена».
- **Распечатать заявку** → открывается печатная версия (PDF или print-preview).
- **Назначить волонтёра** *(v2)* → привязать конкретного волонтёра к заявке.
- **Отметить «Выполнено»** *(v2)* → финальный статус после оказания помощи.

---

### 5. Печать / экспорт

**Печатная карточка заявки** (формат A4, одна страница):
```
Заявка № ____
Дата: __________
ФИО: _________________________________
Район: _________________________________
Необходимая помощь:
  [текст]

Количество необходимых волонтёров: ___
Комментарий:
  [текст]

Подпись волонтёра: ______________________
Дата выполнения: ________________________
```

**Экспорт заявок** *(v2)*:
- Экспорт списка заявок в Excel (.xlsx).
- Экспорт списка заявок в PDF.

---

### 6. Справочник районов

Список районов хранится в базе данных (справочная таблица). Администратор может управлять справочником через стандартную Django admin (добавлять, редактировать, удалять районы). Форма заявки использует справочник для поля «Район» (выпадающий список).

---

### 7. Персональные данные и согласие (ПДн)

- На форме должно быть явное согласие на обработку персональных данных (чекбокс или текстовая оговорка со ссылкой на политику).
- Политика конфиденциальности и текст согласия оформляются в соответствии с законодательством.
- *(v2)* Если добавляется сбор контактных данных (e-mail, телефон) — реализовать соответствующее расширение политики ПДн и формы согласия.

---

### 8. Уведомления *(v2)*

- Автоматическая отправка уведомления заявителю после принятия или отклонения заявки.
- Каналы: электронная почта и/или SMS (при наличии контактных данных в заявке).
- Архитектура должна позволять подключить уведомления без переработки основного кода.

---

## Нефункциональные требования

### Технологический стек

| Слой | Технология |
|------|-----------|
| Backend | Django (Python) |
| Frontend | HTML + CSS + JavaScript (серверный рендеринг), адаптивная вёрстка |
| База данных | PostgreSQL |
| Контейнеризация | Docker + Docker Compose |
| Деплой | rsync + docker-compose на VPS (Linux) |

### Интерфейс

- Современный минималистичный дизайн.
- Адаптивная вёрстка: компьютер, планшет, телефон.
- Простая и понятная административная панель.

### Надёжность и безопасность

- Авторизация для доступа к административной панели.
- Валидация обязательных полей (клиент + сервер).
- Защита от спама на форме (reCAPTCHA или аналог).
- Хранение секретов через переменные окружения (`.env`), не в репозитории.

### Расширяемость

- Модель данных и архитектура должны допускать добавление поиска, фильтрации, назначения волонтёров, экспорта и уведомлений в следующих версиях без переписывания ядра системы.

---

## Модель данных (заявка)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | integer, PK | Уникальный номер заявки |
| `full_name` | string | ФИО заявителя |
| `district` | FK → District | Район |
| `description` | text | Описание необходимой помощи |
| `volunteers_needed` | integer, nullable | Количество необходимых волонтёров |
| `comment` | text, nullable | Дополнительный комментарий |
| `status` | enum | Новая / Принята / Отклонена |
| `created_at` | datetime | Дата и время создания |
